# Nexus Controlled Low-Risk Renderer Hidden Standard User Mount Point Contract

## 1. Purpose and Scope

Phase 13H defines the contract for a future hidden Standard User mount point for the controlled low-risk renderer. This is a contract and QA guard phase only.

This phase does not add the mount point to `public/index.html`, does not wire the renderer into `public/app.js`, does not activate the default-off visible feature flag, and does not change Standard User runtime behavior.

The future mount point name reserved by this contract is:

`nexus-controlled-low-risk-renderer-root`

## 2. Relationship to Prior Renderer Phases

This contract builds on the pre-activation stack:

- Phase 12Y: controlled runtime flag-on test harness implementation.
- Phase 12Z: controlled runtime flag-on harness browser regression validation.
- Phase 13A: visible UI design contract.
- Phase 13B: inert DOM prototype behind test fixture only.
- Phase 13C: inert DOM browser regression and contract enforcement.
- Phase 13D: test-only visual snapshot fixture.
- Phase 13E: Standard User readiness review before visible activation.
- Phase 13F: default-off visible feature flag design.
- Phase 13G: Standard User validation after the renderer pre-activation stack.

Phase 13H prepares a future DOM location contract, but the actual Standard User DOM mount remains absent until a later explicit implementation phase.

## 3. Current Phase Decision

The hidden mount point is intentionally not added in this phase. `public/index.html` must not contain `nexus-controlled-low-risk-renderer-root`, `data-nexus-controlled-low-risk-renderer-root`, or an equivalent controlled low-risk renderer root.

The renderer must remain inactive, unwired, and invisible in the Standard User build.

## 4. Future Mount Point Contract

When a future phase adds the mount point, it must follow these rules:

- The mount point must be absent by default until a future explicit implementation phase.
- The mount point must be hidden and default-empty unless all renderer gates pass.
- The mount point must not render anything on page load by itself.
- The mount point must not become toggleable through query parameters.
- The mount point must not become toggleable through `localStorage`.
- The mount point must not become toggleable through `sessionStorage`.
- The mount point must not become toggleable through user-controlled DOM edits.
- The mount point must not create visible UI without the literal boolean flag `enableControlledLowRiskRendererVisibleUi === true`.
- The mount point must not render if the flag is missing, false, null, malformed, or a truthy string.

## 5. Future Required Gates

Future rendering into the mount point may occur only when all of these gates pass:

- `enableControlledLowRiskRendererVisibleUi === true`.
- The renderer category is explicitly allowlisted.
- The action is low-risk only.
- Execution is false.
- Provider handoff is false.
- Permission request is false.
- Navigation is false.
- Call or message behavior is false.
- Payment or marketplace transaction behavior is false.
- Camera, location, health mutation, emergency, account, and identity behavior are false.
- The renderer uses text-only DOM insertion.
- Raw HTML is rejected.
- The rendered content is review-only and states: "No action has been taken."

## 6. DOM Safety Rules

The future mount point must remain an inert display surface. The first visible phase must not add:

- Action buttons.
- External links.
- Navigation links.
- Click handlers.
- Network calls.
- Storage writes.
- Confirmation modals.
- Permission prompts.
- Provider handoff.
- Native bridge calls.
- Route changes.
- Workflow auto-open behavior.
- Execution, staging, or dispatch behavior.

The only future state mutation allowed by this contract is a separately approved visible review-only rendering state used to show or clear inert low-risk preview content.

## 7. Explicitly Prohibited Behavior

The mount point must never make metadata authoritative. These fields and objects must not execute actions by themselves:

- `selectedToolId`.
- `agentAction`.
- `plannedActions`.
- `policyDecision`.
- `sessionMemory`.
- Low-risk suggestion metadata.
- Controlled action metadata.

The mount point must not unlock provider, call, message, payment, location, camera, marketplace transaction, account/profile, health, telehealth, emergency, identity, or native bridge behavior.

## 8. Standard User Demo Posture

Standard User remains the primary validation target. Until a later visible activation phase, the Standard User demo must behave exactly as before:

- No visible controlled low-risk renderer mount point.
- No hidden/debug-only metadata exposed.
- No visible preview card generated by the new mount contract.
- No route, permission, provider, confirmation, or execution changes.
- Existing low-risk suggestion labels and existing controlled preview behavior remain unchanged.

## 9. QA Expectations

The Phase 13H QA guard must verify:

- This document exists and contains the hidden mount point contract.
- `public/index.html` does not contain the future mount point.
- `public/app.js` does not query or render into the future mount point.
- `server.js` does not expose a mount point or visible renderer activation endpoint.
- The default-off visible feature flag design remains documented.
- Prior controlled low-risk renderer QA guards remain included in the Nexus Workforce suite.
- The new Phase 13H guard is included in the Nexus Workforce suite.
- The package alias `qa:nexus-controlled-low-risk-renderer-hidden-standard-user-mount-point-contract` exists.

## 10. Recommended Next Phase

The recommended next phase is a browser-validation or implementation-readiness checkpoint before any Standard User mount point is added.

A future implementation phase may add the hidden empty mount point only after this contract remains green and the Standard User build is manually validated again. Even then, the mount point should be empty by default and unable to render content unless the approved visible feature flag and all low-risk gates pass.
